# [heat-2] spec.md — 答え合わせ用

**注意: kata.ts を実装してから開くこと。**

## 想定仕様

① `new Stock(quantity, limit)`:
- 負の値は例外
- `quantity > limit` は例外(不正な組み合わせを許さない)

② `add(amount)`: 上限を超える追加は例外。元の Stock は変更しない。

③ `reserve(amount)`: 在庫不足は例外。元の Stock は変更しない。

④ `canAdd / canReserve`: 例外を投げず boolean を返す。事前判定用。

⑤ `isEmpty`: `quantity === 0` のとき true。

⑥ `add / reserve` への負の amount は例外(`canAdd / canReserve` は単に false を返す)。

## 自分の契約と比較する観点

### 「不正な組み合わせ(quantity > limit)を拒否する」設計

heat-1 の Price と同じく、**完全コンストラクタ**の応用。

```ts
if (quantity > limit) {
  throw new Error(...);
}
```

これにより、**「Stock インスタンスが存在する」=「在庫数は上限以下である」** が保証される。

この保証がないと、他のメソッドが防御的にチェックする必要が出てくる:

```ts
// 完全コンストラクタでない場合
add(amount) {
  if (this.quantity > this.limit) {
    // 不正な状態から始まっている可能性を考慮しないといけない
  }
}
```

**不正な状態を持ったオブジェクトをそもそも作れなくする**ことで、メソッド側のロジックが単純になる。これがミノ駆動本でいう **「不正状態から防護する」** という設計姿勢。

### 「can〜 と 実行系」を両方持つ設計

```ts
if (stock.canAdd(10)) {
  stock = stock.add(10);
}
```

vs

```ts
try {
  stock = stock.add(10);
} catch { ... }
```

想定仕様では **両方持たせている**。理由:

- `canAdd` は **「UI ボタンを有効にしていいか」** のような **判定だけしたい場面** で使う
- `add` は **「実行の意思がある」** ことを表す。失敗は例外を投げる(=異常事態)

つまり **「問い合わせ(query)」と「実行(command)」を分ける** という設計。これは次の forge-mino-02 で深掘りする予定のテーマ(コマンド・クエリ分離)の入口。

ただしトレードオフあり: `canAdd` が `true` でも、別スレッド・別リクエストで状態が変わって `add` が失敗する可能性は残る。**「事前判定が真でも実行が必ず成功するとは限らない」** のは、Web アプリでは常に意識すべき点。

### 「不変性」の再確認

heat-1 と同じ判断: `add / reserve` は新しい Stock を返す。

理由も同じ:
- 関数の引数として渡した Stock が、関数の中で勝手に変わらないことを保証する
- メソッドチェーンが書ける

ただし「在庫」という業務概念は、本来は**可変**(現実世界の在庫は減ったり増えたりする)。それでも不変オブジェクトとして扱う理由は:

- **「ある時点での在庫の状態」を値として扱える**
- 履歴を残したいとき、各状態がそのまま使える
- テストで「この時点ではこうだった」と固定して検証できる

DB に永続化するときは別の責務(Repository など)が引き受ける。「**メモリ上の表現は不変、永続化は別レイヤー**」という分け方。

### 「Stock が limit を持つ」設計

なぜ Stock 自身が limit を持つか:

- もし「商品マスタが limit を持ち、Stock は quantity だけ」だと、`add` を呼ぶたびに **限度を外から渡す** ことになる
- すると `addStock(stock, limit, amount)` という署名になり、低凝集に逆戻りする
- 「在庫数」と「在庫上限」は **常にペアで現れる** → だから同じクラスに置く

これがミノ駆動本でいう **「データクラスにせず、振る舞いとデータを揃える」** ということ。

ただし注意: `limit` が動的に変わる業務(例: 季節で上限を変える)では、別の設計が必要になる。今回は「上限は商品ごとに固定」を前提とした設計。**仕様によって妥当な設計は変わる**。

## 観察ポイント: プリミティブ2個 vs 値オブジェクト1個

problem.md の問い「シグネチャの違い」への答え:

```ts
// プリミティブ2個
function canAdd(stock: number, limit: number, amount: number): boolean
// 呼び出し
canAdd(item.stockLimit, item.stock, 10);  // 順番を間違えた! IDE は気付かない
```

```ts
// 値オブジェクト
function canAdd(stock: Stock, amount: number): boolean
// 呼び出し
canAdd(item.stock, 10);  // 順番を間違えようがない
```

**同じ型が並んでいる引数リストは、間違える余地を埋め込んでいる**。値オブジェクトに束ねると:

- 引数の順番ミスが構造的に起きない
- IDE がオートコンプリートで助けてくれる
- 型を見ただけで「何のための関数か」が分かる

これがミノ駆動本でいう **「目的駆動名前設計」と「プリミティブ型執着の解消」が連動する** という話。

## 観察ポイント: heat-2 で「外から見えなくなったもの」

| heat | 何が見えなくなったか |
|------|--------------------|
| heat-1 | 計算ロジック、端数処理、表示フォーマット |
| heat-2 | **プリミティブの組み合わせ方**(quantity と limit の関係) |

heat-1 では「ひとつのデータに対する操作」を隠した。heat-2 では「**複数データの不変条件**」(quantity ≤ limit) を隠した。

呼び出し側は「在庫を扱う」とだけ書けば、「上限を意識する」「ペアで渡す」ことから解放される。

次の heat-3 では、もう一段「**コレクションそのもの**」が見えなくなる。

## 観察ポイント: heat-1 → heat-2 の進化

| 項目 | heat-1 (Price) | heat-2 (Stock) |
|------|---------------|----------------|
| 隠したもの | 計算ロジック | 複数プロパティの関係 |
| 不変条件 | 「値が負でない」 | 「quantity ≤ limit」 |
| 操作 | withTax / discount | add / reserve |
| 問い合わせ | なし(toNumber のみ) | canAdd / canReserve / isEmpty |

heat-1 は **「振る舞いをデータの近くに置く」** だけだった。heat-2 は **「振る舞い + 不変条件 + 問い合わせ」** をまとめて持つ。これが値オブジェクトとして一段成熟した形。
