# [heat-1] spec.md — 答え合わせ用

**注意: kata.ts を実装してから開くこと。**

## 想定仕様

① `new Price(amount)`: 負の金額は例外を投げる。0 は許容。

② `withTax(rate)`: 元の Price は変更せず、新しい Price を返す。負の税率は例外。

③ `discount(rate)`: 元の Price は変更せず、新しい Price を返す。`[0, 1]` 範囲外は例外。

④ 端数処理は `Math.floor`(切り捨て)。

⑤ `toDisplay()`: `¥1,200` のような日本円表記。

⑥ `toNumber()` は内部の amount を返す(エスケープハッチ)。

## 自分の契約と比較する観点

`problem.md` の4問それぞれについて、JSDoc に書けていたか確認する。特に【設計の読解】に「なぜ Price を返すのか」「なぜ getter ではなく toNumber() か」が書けていたかを見る。

### 「不正な値はコンストラクタで弾く」設計

ミノ駆動本の **「完全コンストラクタ」** の考え方。

```ts
constructor(amount: number) {
  if (amount < 0) throw new Error(...);
  this.amount = amount;
}
```

これにより、**「Price インスタンスが存在する」=「妥当な価格である」** が保証される。型システムだけでは表せない制約を、コンストラクタで担保する。

逆の設計(妥当性を呼び出し側に任せる)だと、どこで `if (price >= 0)` を書き忘れたかをコードベース全体に散らばせることになる。**バグの原因がデータの発生源から離れる**。これが低凝集の本質。

### 「メソッドが Price を返す」設計

`withTax(rate): Price` の戻り値が `number` だったら何が困るか:

```ts
// もし number を返したら
const taxIncluded: number = price.withTax(0.1);
// 呼び出し側は「これは税込か? 割引後か?」を文脈から推測する必要がある
// さらに、taxIncluded * 0.5 のような「価格らしくない演算」が書けてしまう
const broken = taxIncluded * 0.5;  // これは何?
```

Price を返すことで:
- 戻り値が「価格である」ことが型レベルで保証される
- `* 0.5` のような無意味な演算ができなくなる
- 次の操作(さらに割引、表示)もメソッドチェーンで自然に書ける

これがミノ駆動本でいう **「プリミティブ型執着(Primitive Obsession)」からの脱却**。

### 「不変性」の設計

`withTax` で元の `Price` を書き換えない選択。

書き換える設計だとどうなるか:

```ts
// もし mutable だったら
const price = new Price(1000);
calcDisplay(price);  // この中で price.withTax(0.1) が呼ばれていたら
console.log(price.toNumber());  // 1000 ? 1100 ? 読まないと分からない
```

不変にすると、**「引数として渡した値が、関数の外側で勝手に変わらない」** ことが保証される。これは「関数を疑わずに呼べる」という巨大な認知的恩恵をもたらす。

### 「toNumber() で取り出す」設計

なぜ `price.amount` を public にせず、`price.toNumber()` を経由させるのか:

- public フィールドだと、外から `price.amount = -100` で破壊できてしまう(`readonly` でも、JS ランタイムでは保証されない)
- メソッド経由なら、**「取り出す」という意思を持った操作**として記録される
- 将来「実は内部表現を整数の銭単位に変えたい」となったとき、メソッドなら隠せる(public フィールドだと全呼び出し箇所が影響を受ける)

これは **「データを直接触らせず、操作を提供する」** という原則。Tell, Don't Ask の入口。

### 端数処理の方針

`Math.floor` を選んだのは、EC 文脈で **「お客様の不利にならない方向」** に倒すため。

- 税込計算で切り上げると、表示価格 > 実際の請求になり得る
- 商品単価では切り捨てが多数派

ただし業界・契約次第。**端数処理は仕様であって実装の都合ではない**。問題文に書いていないなら、自分で決めて JSDoc に明記する。これが【契約】の4問目「暗黙の決め」に該当する。

## 観察ポイント: 「関数集約」と「クラス化」の本質的な違い

problem.md の問い「`priceUtils.calcTax(price, rate)` と `new Price(price).withTax(rate)` の違いは何か」への答え:

| 観点 | 関数集約 | クラス化 |
|------|---------|---------|
| データの型 | `number`(価格かどうか不明) | `Price`(価格であることが型で保証) |
| 不正値の混入防止 | 各関数で都度チェック | コンストラクタで一箇所 |
| 関連操作の発見 | grep が必要 | IDE がメソッド一覧を出してくれる |
| 内部表現の変更 | 全呼び出し箇所が影響 | クラス内に閉じる |

関数集約は **「ファイルが整理されているだけ」** で、**型システムの保護を受けていない**。クラス化は **「データそのものに意味を持たせる」** ことで、コンパイラを味方につける。

これがミノ駆動本でいう **「データクラス」 vs 「自分で考えるクラス」** の差。データクラス(public な値だけのクラス)は本質的に関数集約と同じ。

heat を通じて、外から見えなくなるものが階段状に増えていく。heat-1 の段階では:
