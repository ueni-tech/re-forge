# [heat-3] spec.md — 答え合わせ用

**注意: kata.ts を実装してから開くこと。**

## 想定仕様

① `new OrderItems(items?)`: 初期配列を **防衛コピー** して保持。

② `add(item)`: 上限到達は例外。元のインスタンスは変更せず新しいインスタンスを返す。

③ `totalAmount`: 空のとき 0。`price * quantity` の合計。

④ `count`: 内部配列の長さ。

⑤ `hasHighPriceItem`: 5000円以上の単価が含まれるか。空のときは false。

⑥ `toSnapshot`: `readonly OrderItem[]` 型で返す。型レベルでの保護。

## 自分の契約と比較する観点

### 「防衛コピー」の設計

```ts
constructor(items: OrderItem[] = []) {
  this.items = [...items];  // 防衛コピー
}
```

なぜコピーするか:

```ts
const items = [item1];
const orderItems = new OrderItems(items);
items.push(item2);  // 防衛コピーがなければ、内部の配列も変わってしまう
```

外部から渡された参照を **そのまま信じない**。これは「**不変オブジェクトを守るための入り口での義務**」。

ただしトレードオフあり:
- コピーはコスト(浅いコピーでも O(n))
- 大量データだと無視できない場合がある

実務では「呼び出し側が変更しないと約束できる」場合に限り、コピーを省略することもある。**約束は型では表せないので、JSDoc に書く**。これがまさに【契約】の役割。

### `toSnapshot()` で `readonly OrderItem[]` を返す設計

選択肢は3つあった:

| 選択肢 | 何が守られる | 何が守られない |
|--------|-----------|-------------|
| `items` をそのまま返す | 速い | 呼び出し側が push できる |
| `[...this.items]` でコピー返却 | ランタイムでも保護 | コスト・要素オブジェクトは共有 |
| `as readonly OrderItem[]` で返す | TS が変更を弾く | ランタイムでは通る |

想定仕様は3番目。理由:

- TypeScript プロジェクトなら、**型レベルで防げれば実質十分**
- ランタイムで防ぐコストを払うほどの脅威ではない(攻撃者ではなく、うっかりミスへの防御)
- パフォーマンスを犠牲にしない

「**型レベルの保護をどこまで信頼するか**」という判断。チームのコーディング規約による。

注意点: `readonly OrderItem[]` は「**配列を変更できない**」だけで、「**要素オブジェクトのプロパティを変更できない**」わけではない。

```ts
const snapshot = orderItems.toSnapshot();
snapshot[0].quantity = 999;  // これは通ってしまう
```

完全に守りたければ `readonly OrderItem` で要素も readonly にする。今回はそこまでしていない。**「どこまで守るか」も設計判断**。

### 「MAX_ITEMS を static で持つ」設計

```ts
static readonly MAX_ITEMS = 10;
```

なぜインスタンス変数ではなく `static` か:

- インスタンスごとに上限が変わらないなら、static が自然
- `OrderItems.MAX_ITEMS` で外から参照できる → UI 側で「あと何個追加できる」表示に使える
- インスタンスごとに上限を変えたい設計なら、コンストラクタ引数で受ける

もし将来「VIP 会員は上限20」のような業務になったら、設計を変える必要がある。**今の業務に合った設計を選ぶ**。先回りして柔軟性を埋め込みすぎると、複雑さだけが残る。

### 「閾値(5000)を定数として外に出す」設計

```ts
const HIGH_PRICE_THRESHOLD = 5000;
```

メソッド内のマジックナンバーでもなく、コンストラクタ引数でもなく、ファイル定数。

- マジックナンバーだと「5000の意味」がコードから読み取れない
- コンストラクタ引数だと、呼び出し側に閾値を意識させてしまう(ビジネスルールが外に漏れる)
- ファイル定数なら、**ビジネスルールがクラスの責任範囲に閉じる**

ただし、もし「閾値が頻繁に変わる」「環境ごとに違う」業務なら、別の場所(設定ファイル・DB)に置くべき。**「変わりやすさ」が設計を決める**。

### `add` が新しいインスタンスを返す設計

heat-1, heat-2 と同じ。理由も同じ。

メソッドチェーンで使うことを想定していないとしても、**「同じインスタンスを参照している複数の箇所が、知らないうちに状態を共有しない」** という保証は大きい。

## 観察ポイント: 「便利さを捨てて何を得るか」

problem.md の問いへの答え:

**捨てたもの:**
- `array.filter().map()` の自由な変形
- `for` ループや `push` の直感的な操作
- 「配列だから何でもできる」という万能感

**得たもの:**
- **不変条件の保証**(MAX_ITEMS の遵守)
- **業務用語のメソッド**(`hasHighPriceItem`)が一箇所に集まる
- **「外から何ができるか」が型で明示される**

これは **「自由を制限することで、意図を明確にする」** という設計姿勢。これがファーストクラスコレクションの本質。

ミノ駆動本の言葉で言えば: **「コレクション操作のロジックを、コレクションを保持するクラスに凝集させる」**。

「半年後にこのコードを読む自分」は、`orderItems.hasHighPriceItem()` を見ただけで意図を読み取れる。`order.items.some(i => i.price >= 5000)` だと、`5000` の意味を毎回考える必要がある。**短期の便利さと長期の可読性のトレードオフ**。

## 観察ポイント: heat-3 で「外から見えなくなったもの」

| heat | 何が見えなくなったか |
|------|--------------------|
| heat-1 | 計算ロジック、端数処理、表示フォーマット |
| heat-2 | プリミティブの組み合わせ方(quantity ≤ limit の不変条件) |
| heat-3 | **配列そのもの**、業務的な判定基準(5000円という閾値) |

heat-3 では「**何個入っているか以外、外からは何も操作できない**」状態を作った。これは **「Tell, Don't Ask」** の徹底:

- **Ask**(問い合わせて自分でやる): `if (order.items.some(...)) { ... }`
- **Tell**(やってほしいことを伝える): `if (order.items.hasHighPriceItem()) { ... }`

呼び出し側は **「何を判定したいか」を業務語彙で伝える** だけ。判定ロジックは中で動く。

## 観察ポイント: 3 heat を貫く軸

| heat | データ | 操作の所在 (低凝集 → 高凝集) |
|------|--------|----------------------------|
| heat-1 | 1つの値 | 関数群 → クラス内メソッド |
| heat-2 | 関連する複数値 | プリミティブ引数の関数 → 値オブジェクトのメソッド |
| heat-3 | コレクション | 配列の生操作 → コレクションクラスのメソッド |

一本の問いを思い出す: **「データを操作する権利を、誰が持つべきか」**

3 heat すべての答えは同じ: **「データを保持しているクラス自身」**。

低凝集の解消とは、操作の権利を **データの近くに集約していく** プロセス。クラス化はその手段。

そしてこれは **カプセル化** の本質でもある。各 heat で「外から見えなくなったもの」が階段状に増えていったのは、その表れ。
