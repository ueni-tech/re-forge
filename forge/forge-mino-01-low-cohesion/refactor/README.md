# forge-mino-01-low-cohesion / refactor — 既存コードからの抽出訓練

forge-mino-01 と同じテーマ(Price / Stock / OrderItems)を、**既存の散らかったコードからリファクタする**演習。

forge 系(`../heat-N/`)が「白紙からクラスを実装する」演習だったのに対し、refactor 系は **「動いている既存コードを、振る舞いを変えずに構造化する」** 演習。

## forge 系との違い

| 観点 | forge 系 (`../heat-N/`) | refactor 系 (`./heat-N/`) |
|------|----------------------|---------------------------|
| 開始状態 | 白紙の kata.ts | 動いている既存コード |
| API シグネチャ | problem.md で**指定済み** | csk が**設計する** |
| 主な訓練 | 不変条件と振る舞いの実装 | 抽出範囲の判断・API 設計・呼び出し側の書き換え |
| 安全網 | テストを通せば完了 | **既存テストを緑のまま保ちながら**段階的に変える |

## refactor 系を貫く軸

> **「既存コードのどこを切って、どう抽出するか。そして抽出した後、呼び出し側はどう書き換わるか」**

forge 系が「データを操作する権利を、誰が持つべきか」を扱ったのに対し、refactor 系は **「現状の散らかった権利分配を、どう整理するか」** を扱う。

## refactor の鉄則: 振る舞いを変えない

各 heat には `before/` と `kata/` がある。`before/` はリファクタ前の動くコード。`kata/` は csk が refactor する作業場所(初期状態は `before/` のコピー)。

**`kata.test.ts` は `before/` 時点で全て pass する**。csk がリファクタしても、外から見た振る舞いが変わらなければテストは緑のまま。

これが Martin Fowler の Refactoring の鉄則:

> **リファクタリングとは、外部から見た振る舞いを変えずに、内部構造を改善すること。**

テストが緑のまま、構造だけが改善されていく。テストが赤になったら、それは refactor ではなく**仕様変更**。

## heat 構成

| heat | 抽出対象 | テーマ | 対応する forge 系 |
|------|---------|-------|--------------------|
| heat-1 | `Price` | 散らばった価格計算ユーティリティ群を1つの値オブジェクトに | forge-mino-01 heat-1 |
| heat-2 | `Stock` | プリミティブ2-3個渡しの在庫関数群を値オブジェクトに | forge-mino-01 heat-2 |
| heat-3 | `OrderItems` | 配列を直接操作するコードをファーストクラスコレクションに | forge-mino-01 heat-3 |

## 各 heat のディレクトリ構成

```
heat-N/
├── problem.md      # 症状と「やってはいけないこと」、設計の問い
├── before/         # リファクタ前のコード(動く・触らない)
├── kata/           # csk の作業場所(初期状態は before のコピー)
├── kata.test.ts    # before の振る舞いを保証するテスト(kata/ に対して走る)
├── solution/       # 模範解答
└── spec.md         # 答え合わせ + 呼び出し側 before/after diff
```

## 進め方

1. `problem.md` を読む。「現状の症状」と「あなたが決めること」を理解する
2. `before/` のコードを読む。**ただし**`kata/` は触らない
3. `kata/` のコードを読み、何をどう抽出するか頭の中で設計する
4. `kata/` を**少しずつ**書き換える。各ステップで `npx vitest forge-mino-01-low-cohesion/refactor/heat-N` を走らせて緑を保つ
5. 完了したら `solution/` と `spec.md` を開いて答え合わせ

```bash
npx vitest forge-mino-01-low-cohesion/refactor
```

## 推奨する進め方の順序

1. 先に**forge 系(`../heat-N/`)を解く**ことを推奨する。白紙実装で身につけた「クラスの形」が、refactor 系での抽出ターゲットになる
2. forge 系を解き終えた後で refactor 系に進むと、**「ゴール像が見えている状態でリファクタする」**経験ができる
3. 同じテーマを2回経験することで、**「設計とは何か」が立体化**する

ただしこれは推奨であって強制ではない。csk が「先に refactor を見たい」と判断したらそれでもよい。判断の理由を言語化することが、シニアエンジニアへの一歩。
