# [heat-2] TTL 付きキャッシュ

## 実務での使われ方

API レスポンスや権限情報など、しばらくは再利用したいが永続キャッシュにはしたくないデータを、時間経過後に自動で無効化したいときのパターン。

heat-1 のキャッシュに「時間軸」を加える。

## やりたいこと

エントリが指定ミリ秒経過後に**自動的に無効化される**キャッシュを作る。

heat-1 と異なり、`delete` は外部に公開しない。期限管理はキャッシュ自身の責任。

## 入出力

```ts
function createTTLCache<V>(ttl: number): {
  get: (key: string, factory: () => V) => V;
}
```

- **createTTLCache(ttl)**: ttl はエントリの有効期限(ミリ秒)。
- **get(key, factory)**: heat-1 と同様の API。ただし期限切れエントリへの get では factory が再実行される。

## あなたが決めること

実装する前に、自分で以下を決めて JSDoc の【契約】と【設計の読解】に書く:

### 期限の表現

- 「いつ作られたか」を保存する? 「いつ切れるか」を保存する?
- どちらでも実装できるが、判定コードの簡潔さに差が出る。比較してみる。

### 境界の扱い

- `ttl = 1000` で保存した値が、`Date.now() === expiresAt` の瞬間に取りに来たら有効? 期限切れ?
- どちらでも仕様として成立するが、**決めておかないと挙動がブレる**。

### ttl をどこに渡すか

- なぜ `createTTLCache(ttl)` で固定し、`get(key, factory, ttl)` にしていないのか?
- 自分なりの仮説を立てる。

### 別キーの独立性

- キー A の TTL とキー B の TTL は独立している? それとも「インスタンス共通の時計」がある?

## 設計の問い(実装前に考える)

このheatの本質は **「ポリシーをどこに固定するか」** という設計判断にある。

- ttl をインスタンスに固定するメリットは何か
- 呼び出しごとに ttl を変えられる柔軟性を失う代わりに、何を得ているか
