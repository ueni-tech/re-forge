# [heat-2] spec.md — 答え合わせ用

**注意: kata.ts を実装してから開くこと。**

## 想定仕様

① `Date.now() < expiresAt` のとき保存値を返す(factory は呼ばない)。

② エントリが無い、または期限切れなら `factory()` を1回呼び保存して返す。

③ 境界: `Date.now() === expiresAt` の瞬間は**期限切れ扱い**。

④ 別キーの期限は独立している。

## 自分の契約と比較する観点

### 期限の表現: 絶対時刻 vs 経過時間

想定仕様は **絶対時刻 `expiresAt`** を採用している。

```ts
type Entry<V> = { value: V; expiresAt: number };
```

もし「作成時刻 `createdAt`」を保存していたら、判定はこうなる:

```ts
if (Date.now() - entry.createdAt < ttl) { ... }
```

`expiresAt` なら:

```ts
if (Date.now() < entry.expiresAt) { ... }
```

後者の方が単純。**「判定のたびに計算する」より「保存時に1回計算して保存しておく」** ほうが、読みやすく、ttl が外から変わる事故も防げる。

### 境界の扱い

`Date.now() === expiresAt` を期限切れ扱いにする選択は、`<` 演算子で済むという実装の単純さも理由のひとつ。`<=` だと「ぴったりの瞬間は有効」という意味になる。

これは「**仕様の選択が実装の単純さに影響する**」例。仕様を決めるとき、実装の都合も判断材料に入る。

### ttl をインスタンスに固定する設計

`createTTLCache(ttl)` で固定する判断のメリット:

- 「**このインスタンスはこの寿命**」というポリシーが型レベルで明示される
- 呼び出し側で ttl をバラつかせる事故を防ぐ
- 用途ごとに異なるインスタンスを作る運用が自然になる

```ts
const userCache = createTTLCache<User>(60_000);    // 1分
const configCache = createTTLCache<Config>(3600_000); // 1時間
```

柔軟性を捨てて、**ポリシーの明示性**を取った設計判断。

### 期限切れエントリの後始末

期限切れエントリを `delete` してから `set` するか、いきなり `set` で上書きするか。

想定仕様では上書き。理由は `delete` を挟む実利がなく、Map 操作が1回減るため。

ただし**この判断は「メモリ削減」の観点だと不十分**。期限切れになって以降、一度もアクセスされないキーは Map に残り続ける。LRU や定期的なクリーンアップが必要なケースでは、別の設計が要る。

## 観察ポイント: heat-1 との差分から学ぶ

| 項目 | heat-1 | heat-2 |
|------|--------|--------|
| キャッシュの無効化 | 呼び出し側が `delete` する | 時間が無効化する |
| 状態 | 値のみ | 値 + 期限 |
| API | get / delete | get のみ |
| ポリシーの所在 | 呼び出し側 | インスタンス |

heat-1 から heat-2 への変化は **「無効化の責任を、呼び出し側からキャッシュ自身に移した」** という設計上の変化。

これにより:
- 呼び出し側のコードが簡潔になる(delete を忘れる事故が減る)
- 一方で「いつ無効化されるか」を呼び出し側がコントロールできなくなる

責務を引き受けることと、柔軟性を捨てることはセット。**設計判断はトレードオフ**。
