# [heat-3] spec.md — 答え合わせ用

**注意: kata.ts を実装してから開くこと。**

## 想定仕様

① `validKeys` に含まれない key: **何もしない**(callback / storage / fetcher のいずれも触らない、例外も投げない)。

② キャッシュヒット: fetcher を呼ばず `callback(value)` を**同期で** 1 回。

③ キャッシュミス: fetcher を呼び、完了時に `storage.set` してから `callback(result)` を 1 回。

## 自分の契約と比較する観点

`problem.md` の4問（正常時 / 困った入力 / しないこと / 暗黙の決め）それぞれについて、【契約】に書けていたか確認する。DI の理由を【契約】に書いていないかも見る。

### 「許可されていないキーは何もしない」設計

例外を投げず、callback も呼ばない選択。

- **メリット**: 呼び出し側がエラーハンドリングを書かなくて済む。安全側に倒れる。
- **デメリット**: バグでキーが間違っていたとき、無反応で気づきにくい。

「callback を呼ばない」選択は、**「キーが許可されているか分からないコンテキスト」から呼ばれることを想定**している。例えばユーザー入力やイベントから直接キーが渡るケース。

もし「validKeys に必ず含まれる前提」で呼ばれる関数なら、例外を投げる方が正しい。**ユースケースが「無効化されたUIへのクリック」のような確実性の低い入口**だから、無視する設計が選ばれている。

### 「キャッシュヒットは同期で callback」設計

非同期化(`setTimeout(cb, 0)`)する選択肢もあったが、想定仕様は同期。

- **同期の利点**: 呼び出し直後に値が手に入る前提でコードが書ける。UI ループが少ない。
- **同期の欠点**: 「キャッシュヒットだと同期、ミスだと非同期」という**呼び出し側から見て予測できない挙動**になる。Zalgo問題と呼ばれることもある。

トレードオフあり。実務では「常に非同期化」する設計を選ぶこともある。今回は「即座に表示できることを優先」した選択。

### fetcher 完了後の順序: set → callback

逆順だとどうなるか考える:

```ts
// 逆順(問題あり)
_fetcher(key, delay, (result) => {
  callback(result);
  storage.set(key, result);
});
```

callback の中で再度 `load(key)` が呼ばれたら、その時点ではまだ `storage.set` が走っていないため、**2回目の fetch が発火する**。

正しい順序:

```ts
_fetcher(key, delay, (result) => {
  storage.set(key, result);
  callback(result);
});
```

これは「**状態を確定させてから副作用を起こす**」という原則の例。callback が何を引き起こすか分からない以上、callback を呼ぶ前に内部状態を確定させる。

### DI されている3つの依存

- **storage**: 保存先の実装を切り離す。テストでは Map ラッパ、本番では LocalStorage。
- **fetcher**: 既定値 `fakeFetch` を持つので普段は意識しなくていい。テストでは差し替える。
- **validKeys**: ファクトリ引数に固定。許可ポリシーが load 呼び出しごとにブレない。

3つの依存の渡し方が、それぞれ意図を持って違っている:

| 依存 | 渡し方 | 理由 |
|------|--------|------|
| storage | 必須引数 | テストごとに差し替える必要があるため |
| validKeys | 必須引数 | ローダー単位でポリシーを固定するため |
| fetcher | デフォルト値付き | 本番は固定、テストでだけ差し替えたいため |

## 観察ポイント: DI の本当の価値

DIの真価は **「シグネチャが責務を語る」** ことにある。

```ts
function createAsyncCacheLoader(
  storage: Storage,
  validKeys: string[],
  fetcher?: typeof fakeFetch,
): (key, delay, callback) => void
```

このシグネチャを見るだけで:

- 保存先に依存する(storage)
- 許可キーのポリシーを持つ(validKeys)
- 非同期取得手段に依存する(fetcher)

ことが分かる。実装を読まなくても**責務が透けて見える**。

もし内部で `localStorage` と `fetch` を直接呼んでいたら、シグネチャからは依存が見えない。「コードを全部読まないと何が起こるか分からない関数」になる。

**依存を引数化することは、責務を可視化することと同義**。これがDIの本質。

## 観察ポイント: heat-1 → heat-2 → heat-3 の進化

| heat | 主題 |
|------|------|
| heat-1 | クロージャで状態を隠蔽する |
| heat-2 | 時間軸を加え、無効化責任をキャッシュに引き受けさせる |
| heat-3 | 依存を外から注入し、責務を型で明示する |

3つの heat は、**「呼び出し側に意識させない責務を、関数側がどう引き受けるか」** というテーマで一貫している。

- heat-1: 初回かどうか
- heat-2: いつ無効化するか
- heat-3: どこに保存するか、どう取得するか

呼び出し側のコードを単純にするために、関数側が複雑さを引き受ける。これは設計の基本姿勢のひとつ。
