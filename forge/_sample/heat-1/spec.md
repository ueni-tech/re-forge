## 自分の契約と比較する観点

### 「factory を get の引数で受ける」設計

- なぜ `createCache(factory)` ではなく `cache.get(key, factory)` か?
- 前者なら**同じ生成戦略を全キーに適用**することになる。後者なら**キーごとに異なる生成戦略**を使える。
- 「保持」と「生成方法」を分離するという責務の切り分け。

### 「createCache がファクトリ関数である」設計

- もしグローバル変数 `var cache = {}` で書かれていたら何が問題か?
  - テストごとに状態をリセットする手間が増える
  - 複数の独立したキャッシュを持てない
  - 状態の所在が外から見えない
- クロージャに閉じ込めることで、**状態の寿命がインスタンスの寿命と一致する**。

### `has` と `get` の使い分け

- factory が `undefined` を返す可能性を考えるとどうなるか?
- `store.get(key) !== undefined` でチェックすると、「undefined を保存済み」と「未保存」を区別できない。
- `Map.has` を使うとこの区別が可能。**「保存されているか」と「値が何か」を別の問いとして扱う**という設計判断。

## 観察ポイント: クロージャの本質

クロージャは「**関数の外から見えない変数**を、関数の中から触れる」仕組み。

```ts
function createCache<V>() {
  const store = new Map<string, V>();  // 外からは見えない
  return {
    get(key, factory) {
      // store にアクセスできる
    }
  };
}
```

これは「**カプセル化**」をクラスを使わず実現する方法。

- クラス: `private` で見えなくする
- クロージャ: そもそも外に出さない

レガシー jQuery 環境ではクラス構文がない時代から、この技法でカプセル化が実現されてきた。**クラスがなくてもカプセル化はできる**ことを覚えておく。

## 観察ポイント: 「やる/やらない」を get の中で判断する

`if (!store.has(key)) { ... }` という条件分岐は、「**factory を呼ぶかどうかを呼び出し側が知らなくていい**」という設計につながる。

```ts
// 呼び出し側はこれだけ
const config = cache.get("user-config", () => buildExpensiveConfig());
```

呼び出し側は「初回呼び出しなのか、2回目以降なのか」を意識しない。**「呼び出し側に意識させない責務の引き受け方」** はこの forge を通じて何度も出てくるパターン。
