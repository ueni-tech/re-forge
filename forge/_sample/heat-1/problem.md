# [heat-1] キー・バリューキャッシュ

## 実務での使われ方

同一キーに対する重い処理(設定オブジェクトの構築・パース結果など)を1回だけ実行し、以降は共有インスタンスを返したいときのメモ化パターン。

レガシーコードではグローバル変数 `var cache = {}` で書かれがちな処理を、クロージャに閉じ込めてテスト可能にする。

## やりたいこと

「キーに紐づく値」をキャッシュする小さなオブジェクトを生成するファクトリ関数 `createCache` を作る。

値の生成方法はキャッシュインスタンス側ではなく、**呼び出し側が `get` のときに渡す**。

## 入出力

```ts
function createCache<V>(): {
  get: (key: string, factory: () => V) => V;
  delete: (key: string) => void;
}
```

- **createCache**: 引数なし。`get` と `delete` を持つキャッシュオブジェクトを返す。
- **get(key, factory)**: キーに紐づく値を取り出す。値がなければ factory を呼んで得た結果をキャッシュする。
- **delete(key)**: キーに紐づくエントリを削除する。

## あなたが決めること

実装する前に、自分で以下を決めて JSDoc の【契約】に書く:

- 同じキーで2回 `get` を呼んだら、factory は何回呼ばれるべきか?
- `delete` した後に同じキーで `get` を呼んだら?
- 別々のキーで `get` を呼んだとき、互いに干渉する?
- `createCache` を2回呼んで作った2つのインスタンスは、互いに状態を共有する? しない?
- factory が `undefined` を返した場合、「キャッシュ済みで undefined」と「未キャッシュ」をどう区別する?

## 設計の問い(実装前に考える)

このheatの本質は **「状態をどこに置くか」** という設計判断にある。
