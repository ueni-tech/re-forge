# _sample — forge 作成リファレンス

## 新規 forge 作成チェックリスト（TL;DR）

1. **解答の置き場を先に決める** — リポジトリ直下の `README.md` と既存 forge を確認し、`kata.ts` のみスタブにするか、`kata.solution.ts`（など）に解答を分けるかを揃える。
2. **番号付け** — `forge-XX-<theme>/` の `XX` は、既存ディレクトリのうち最大の `forge-XX` の次の連番にする（例: `forge-04` まであれば `forge-05`）。
3. **構成** — 各 heat に `kata.ts` と `kata.test.ts` を置く。
4. **`kata.ts`** — スタブ＋課題コメント。未実装本体は `throw new Error("not implemented")` 等。**仕様の一次ソースはコメント**（テストは固定点）。**各 `export` 関数（スタブ除く）の直上に JSDoc 雛形**を置く（詳細は「kata.ts の JSDoc 雛形」）。
5. **`実務での使われ方:`** — 各 heat の `kata.ts` で、`[heat-N]` タイトル行の**近傍**に必ず書く（詳細は下記）。
6. **暗黙仕様禁止** — 数値・式・順序・DI 契約・非同期の分岐・ミュータビリティの境界は `kata.ts` のコメントに明示する（下記「仕様の明記」）。
7. **`kata.test.ts`** — Vitest。`describe` / `it` は日本語で仕様を述べる。heat あたり **3〜4 本程度**。
8. **heat の漸進** — テーマごとに意味のある段階にする。下記「heat の役割」は**このサンプル（キャッシュ）の一例**であり、すべての forge に当てはまる必然パターンではない。
9. **親 README** — リポジトリ直下の `README.md` の **forge 一覧** に、追加した forge の行を入れる。
10. **検証** — ルートから `npx vitest forge-XX-<theme>` を実行しグリーンを確認する。

## この README の読者と目的（forge 作成用 AI 向け）

**読者:** 新しい `forge-XX-<theme>/` をこのリポジトリに追加する AI。

**親リポジトリ（re-forge）の目的:**  
実務で出会うコードのパターンを **1 テーマずつ切り出し**、学習者が **`kata.ts` を自分の手で実装**して理解を深めるための deliberate practice 用リポジトリです。テストは仕様の固定点として機能し、実装の正しさを検証します。

**forge と heat の意味:**

| 用語 | 意味 |
|------|------|
| **forge** | 1 つの学習テーマ（例: クロージャ、キャッシュ、非同期の世代管理）。ディレクトリ `forge-XX-<theme>/` が 1 forge に対応する。 |
| **heat** | 同一テーマを **抽象度・難易度を段階的に上げた** 練習単位。通常 `heat-1` → `heat-2` → `heat-3` の順で進む。 |

**この `_sample` の役割:**  
「正しい forge の形」の **参照実装・規約の典拠** です。新規 forge を作るときは、ここに示す **ディレクトリ構成・ファイル名・`kata.ts` / `kata.test.ts` の書き方・heat ごとの段階設計** に揃えること。`_sample` 自体はユーザーが解く課題ではなく、AI が新 forge の雛形を生成するときのテンプレート説明書として読む。

**新規 forge を作る AI が達成すべきこと:**

1. **解答の置き場** — 先にリポジトリ直下の `README.md`・既存 forge を確認し、`kata.solution.ts` 等に解答を分ける方針か、`kata.ts` のみスタブかを決めてからファイルを置く。
2. `forge-XX-<theme>/` を追加し（`XX` は既存最大の次番号）、各 heat に `kata.ts` と `kata.test.ts` を置く。
3. `kata.ts` は **スタブ＋課題コメント**（未実装は `throw new Error("not implemented")` 等）まで用意し、**解答コードは学習者向けファイルに混ぜない**（上で決めた方針に従う）。**公開する関数ごとに JSDoc 雛形**（意図・契約・設計の読解・実装上の選択・`@param` / `@returns`・必要なら `@template`）を付ける（下記「kata.ts の JSDoc 雛形」）。
4. `kata.test.ts` は **Vitest** で、**日本語で仕様を記述**した `describe` / `it` と、各 heat で **3〜4 本程度**のテストで要件を固定する。
5. heat の段階は **意味のある漸進** にする（例: 同期最小 → 非同期・時間 → DI）。漸進の組み立てはテーマ依存である。下記「heat の役割」は **キャッシュ `_sample` に即した一例** にすぎず、新規 forge ではテーマに合わせて別の段階設計でよい。
6. **各 heat の `kata.ts` で、`[heat-N]` タイトル行の近傍に「使用想定場面」を書く**（詳細は下記「各 heat の使用想定場面」）。
7. リポジトリ直下の `README.md` の **forge 一覧** に新規行を追加する。

## 仕様の明記（暗黙仕様を作らない）

**目的:** 学習者と次の AI が **テストを逆算する必要なく** `kata.ts` だけで課題を理解できるようにする。テストが要求しているのにコメントに書いていない前提は **暗黙仕様** とみなし、作らない。

**AI が新規 forge / heat を書くときのルール:**

1. **`kata.ts` 上部の課題コメントを唯一の仕様書とする**（ファイル先頭付近。`[heat-N]` タイトル行から続くブロック）  
   テストはその固定点として振る舞う。テストだけが詳しい・コメントが曖昧、という状態にしない。  
   `実務での使われ方:` ブロックは **文脈・動機** の補足であり、①②③ や数式で書かれた **要件本文** が仕様である（下記「各 heat の使用想定場面」参照）。

2. **数値・式・順序は書き切る**  
   遅延時間の決め方、`delay` の計算式、dispatch の順序、`advanceTimersByTime` と整合する前提など、**テストの成否に効く定数・規則はコメントに明示**する。コメントに「例として遅い／早い」だけ書き、実際の ms や式を別にある、は避ける。

3. **スタブ・DI の契約を明示**  
   `fakeLoad` / `fakeRequest` などに **誰がどの引数を決めるか**、完了時にコールバックへ渡る **文字列やデータの形**、古いリクエストで **ユーザー callback を呼ぶ／呼ばない** など、スタブと実装の分割があれば文章で書く。

4. **非同期・エラーの扱い**  
   「棄却された Promise は `reject` か `null` で `resolve` か」「ヒットしなかったときは callback を呼ばないか」など、テストが暗に決めている分岐を仕様に書く。

5. **ミュータブルか・スキップ条件の境界**  
   配列やオブジェクトを **その場で更新するか新しく返すか**、`""` だけを特別扱いするか **空白文字列はどうか**、省略可能な依存は **`undefined` のとき呼ばない** など、テストが検証している境界を文章化する。

6. **純粋関数・reducer などの意味**  
   各 `Action` が state のどのフィールドをどう変えるか、`isValidState` の定義とテストの「各ステップで valid」の関係が一目でわかるように、**型とテストの間に仕様を挟む**。

7. **テスト側のコメント**  
   `kata.ts` に書いた式・数値と **矛盾する短いコメント**（誤った ms のメモなど）をテストに残さない。テスト内の補足は、`kata.ts` の仕様と整合する説明に限定する。

## 各 heat の使用想定場面（必須）

**目的:** 学習者が「この練習が実務のどこに効くか」を早期に把握できるようにする。抽象度の高い API 名だけでは伝わらないため、**heat ごとに** 具体シーンを短く書く。

**AI が新規 forge を作るときのルール:**

1. **各 heat の `kata.ts` に必ず含める**  
   **`[heat-N]` タイトル行の近傍**（課題本文・仕様より前でも後でもよい。ファイル先頭付近にまとまっていればよい）に、見出し **`実務での使われ方:`** を付け、その heat が鍛えているパターンが現れる **具体的な場面** を 2〜4 行で書く。

2. **書く内容の指針**  
   - **どの画面・処理**（例: 商品 PDP のギャラリー、検索のオートコンプリート、カートブロックの表示切替）か、可能なら明示する。  
   - **何を防ぎ・何を達成するコードか**（レースの防止、不正な中間状態の回避、data 属性の安全なパースなど）を一文で添える。  
   - heat の漸進に合わせ、heat-1 はより局所的・単純な場面、heat-3 は複数の関心事が一入口に束ねられる場面、など **難易度と対応させる**。

3. **仕様コメントとの役割分担**  
   「実務での使われ方」は **モチベーションと文脈**。箇条書きの **①②③ の要件・式・境界条件** は従来どおり課題コメント側に書く。両方を混ぜて冗長にしない（場面はイメージ、仕様はテストと一致させる）。

4. **用語**  
   見出しは **`実務での使われ方:`** に統一する（リポジトリ内の既存 forge と揃える）。

## heat の役割（このサンプルテーマにおける一例）

**注意:** heat の漸進は **テーマごとに設計が変わる**。例として同一リポジトリ内では、ギャラリー同期では「シグネチャ → 非同期世代・dispatch」など、キャッシュとは異なる段階の付け方がある。新規 forge では **そのテーマで意味のある段階** を定義し、次節の `_sample` の heat は **参照用の一例** と捉える。

この README が `_sample` で採っている **キャッシュ（cache）テーマ** に即した heat のイメージは次のとおり（下記「このサンプルのテーマ: キャッシュ」と対応）。

| heat | 特徴（サンプルテーマ基準） |
|------|------|
| heat-1 | クロージャで状態を持つ最小実装 |
| heat-2 | 非同期・タイマーを追加（`vi.useFakeTimers()` 使用） |
| heat-3 | DI（依存性注入）を追加し、実装を外から差し替え可能にする |

## kata.ts の規約

```typescript
// [heat-N] 短い説明（1行）
//
// 実務での使われ方:
//   （この heat が想定する利用シーンを 2〜4 行。画面・処理・防ぎたい不具合など）
//
// createXxx() を実装せよ。
// 返却された関数は:
//   ① ...
//   ② ...

// スタブ（変更不要）
export function fakeXxx(...): void { ... }

export type SomeType = { ... };

/**
 * 【意図】呼ぶ側にとって何ができる/楽になるか？（価値を書く。挙動の説明ではなく）
 *  例: ○○を取り出せる。呼び出し側は△△の心配なく扱える。
 *
 * 【契約】何を約束するか？（前提・後条件・エッジ。理由は書かない）
 *  - 前提: 呼ぶ側が守るべきこと
 *  - 後条件: この関数が必ず保証すること
 *  - エッジ: null / 空 / 不一致 / 境界値のとき何をする/しない
 *  例:
 *    - 必ず string を返す
 *    - 該当なしは '' を返す（例外は投げない）
 *
 * 【設計の読解】お題が指定した構造は、なぜその形か？（自分の選択ではなく与えられた構造への推論。自明なら省略）
 *  - なぜこの引数の型か / なぜこの戻り値の型か / なぜこの責務の切り方か
 *  例:
 *    - 引数を最小インターフェース型で受けているのは、テスト容易性のため。
 *    - DI として外から渡す形は、本番とテストで実装を差し替えるため。
 *
 * 【実装上の選択】自分が選んだ実装は、何と迷ってなぜそうしたか？（構造のなぜではなく自分の判断のなぜ。迷いがなければ省略）
 *  - 何と何で迷ったか / なぜこちらを選んだか
 *  例:
 *    - for...of でなく find を採用。「最初の一致を取る」意図が
 *      メソッド名から自明になるため。
 *    - 早期 return を採用。分岐ごとに「使った/使わなかった」が
 *      一目で読めるため。
 *
 * @template V （ジェネリクスがあるとき）
 * @param ... - （記述）
 * @returns （記述）
 */
export function createXxx(...) {
  // ここに実装する
  throw new Error("not implemented");
}
```

### kata.ts の JSDoc 雛形（必須）

**目的:** 学習者が「責務を言語化する」訓練をする。型だけでは伝わらない意図・約束・判断を、関数ごとに 4 つの角度から言語化することで、責務の境界を意識する力を養う。

**責務を 4 つの角度から言語化する:**

| 欄 | 訓練している思考 | 抽象度 |
|----|---------------|--------|
| 【意図】 | **価値**を語る視点 | 利用者の世界 |
| 【契約】 | **約束**を述べる視点 | API の世界 |
| 【設計の読解】 | **構造のなぜ**を推論する視点 | 設計の世界 |
| 【実装上の選択】 | **手段のなぜ**を述べる視点 | 実装の世界 |

上から下へ抽象度が下がる。同じことを別の言葉で繰り返すのではなく、階層が一段ずつ降りていく構造で書く。

**4 見出しと主語のルール:**

| 見出し | 主語 | 書く内容 | 省略可 |
|--------|------|----------|--------|
| 【意図】 | **呼ぶ側** | 何ができる／楽になるか（価値） | 不可 |
| 【契約】 | **API の境界** | 前提・後条件・エッジ（理由は書かない） | 不可 |
| 【設計の読解】 | **お題の設計** | お題が指定した構造のなぜ | 可（自明なら省略） |
| 【実装上の選択】 | **実装した自分** | 自分が選んだ実装のなぜ | 可（迷わなければ省略） |

ファクトリの場合は「呼ぶ側」が「ファクトリ呼び手と戻り値呼び手」の 2 段になる。【意図】はファクトリ呼び手の体験を、【契約】は戻り値（`xxx`）の振る舞いを書く。

**雛形:**

**粒度の目安:**

各欄は **2〜4 行程度** で収まることを目安にする。

- 【意図】1〜2 行（価値の一文）
- 【契約】箇条書きで 2〜4 項目
- 【設計の読解】箇条書きで 1〜3 項目
- 【実装上の選択】箇条書きで 1〜2 項目

書きたいことが多くて 4 項目を超えるなら、**「本当に独立した観点か、重複か」** を一度疑う。重複でなければ書いてよい（re-forge は訓練用なので長くて良い）。

**混ざりやすいパターン:**

各欄が訓練している思考が違うため、書く内容が混ざらないように注意する。

- 【意図】に挙動を書きがち → 「○○を返す」ではなく「○○ができる」
- 【契約】に理由を書きがち → 理由は【設計の読解】【実装上の選択】の担当
- 【設計の読解】に自分の選択を書きがち → 「○○を採用した」は実装上の選択。読解は「お題は○○という形になっている。これは△△のため」
- 【実装上の選択】に構造のなぜを書きがち → 「○○型なのは△△のため」は読解の担当

**AI が新規 forge / heat の `kata.ts` を書くときのルール:**

1. **`export function` / `export async function` ごとに**、宣言の**直上**に上記の雛形を置く。**4 ブロックすべてを置く**（書くことがなければ学習者が削除する運用）。各見出しの **問い・ガイドライン・例文ブロックはそのまま残す**（学習者が埋めながら自分に問えるようにするため）。
2. **学習者が雛形を埋める手順**
   1. 各欄の **1 行目の問い** に答える形で書き出す
   2. **2 行目以降の観点（前提・後条件・エッジなど）** を頭に浮かべて、漏れがないかチェック
   3. 書き終わったら **例文の粒度・トーン** と比べて、自分の文章が冗長すぎないか確認
   4. 埋め終わったら **例文ブロック・ガイドライン・問いの注釈はすべて削除して提出**（完成形は本文のみ）
3. **4 見出しの書き分け**
   - **【意図】** — 呼んだ側の体験を一文で。「**`xxx(...)` を呼べば、…が手に入る／…の心配なく扱える**」の構文。**挙動の説明ではなく目的**を書く。
   - **【契約】** — 前提・後条件・エッジを箇条書きで。**いつ・何回・何が起きる／起きない**。「絶対に呼ばない」「同期で 1 回」など、**型に書けない約束ほど価値が高い**。理由は書かない。
   - **【設計の読解】** — お題が指定した構造のなぜを推論する。例: `InputLike` という最小インターフェース型で受けているのは、jsdom 不要の単体テスト戦略のため。**書けないときは省略してよい**が、省略する前に「本当に自明か」を一度自問する。
   - **【実装上の選択】** — 自分で選んだ実装の根拠。「何と何で迷い、なぜこちらにしたか」を 1 行で。迷う場面がなければ省略してよい。
4. **タグ**
   - ジェネリクスがあるときは **`@template T`**（など）。
   - 引数ごとに **`@param name - （記述）`**。
   - 戻り値があるときは **`@returns`** に短文。型から自明でも、**何が返るか／返した関数の役割**を書く（高階関数なら「`request(key)` を返す」など）。**`void` なら `@returns` を省略してよい。**
5. **スタブ関数（`fakeXxx`）には JSDoc を付けない。** `// スタブ（変更不要）` のコメントだけで十分。
6. **ファイル先頭の課題コメントとの役割分担**
   ①②③ の要件・式・境界は **従来どおり先頭ブロックが一次ソース**。JSDoc は **関数単位の意図・契約・読解・選択の言語化** に寄せ、仕様の全文コピーをしない（冗長にしない）。
7. **`export type` に JSDoc を付けるか**は任意。まずは **動く API の関数**に集中する。

**雛形の進化について:**

この雛形は、学習者が「お題の設計を読解する」段階を主戦場としている。学習者が【設計の読解】を書き続けるうちに、「自分なら別の選択もあり得る」と感じる瞬間が来る。それがフェーズの転換点で、お題側を「設計指示を含まない」形に切り替えていく。雛形は固定するものではなく、学習者の成長段階に合わせて進化させる。

既存の `forge/_sample/heat-*` および各 `forge-XX-*` の `kata.ts` が、この雛形に揃った参照例になる。
