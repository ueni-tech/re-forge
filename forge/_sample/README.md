# _sample — クロージャによるキャッシュパターン

クロージャを使った「状態を関数の中に閉じ込めて、外から API として見せる」パターンを、3段階の複雑度で身につける forge。

**新しい forge を追加するときの雛形** としても使う。AI が forge を生成する際は、リポジトリ直下 [README.md](../../README.md) の「AI向け: 新しい forge を作るときの仕様」と、このディレクトリのファイル構成・セクション構成に従うこと。

## 練習の進め方

各 heat は2段階構成。詳細はリポジトリ直下の README を参照。

1. **段階A**: `problem.md` だけ読んで `kata.ts` を実装する
2. **段階B**: `spec.md` を開いて自分の判断と突き合わせ、テストを通す

```bash
npx vitest forge-01-closure
```

## JSDoc【契約】を書く考え方

【契約】はエッジケース集ではなく、**呼んだ人が実装を読まなくても信じていいこと** を書く。

実装前に4問に答える（各 heat の `problem.md` に当てはめ例あり）。

| # | 質問 |
|---|------|
| 1 | **正常時** — うまくいったとき何が返る？ |
| 2 | **困った入力** — 想定外・境界の入力では？ |
| 3 | **しないこと** — 例外を投げる？ 入力を破壊する？ |
| 4 | **暗黙の決め** — 問題文に無いが自分で決めたことは？ |

仕分け: 型で分かる → 書かない / API の形の理由 → 【設計の読解】 / 内部の呼び順だけ → 【実装メモ】 / 呼んだら結果が変わる → 【契約】

## re-forge のルール（AI が forge を生成するときの注意）

- **合意済み仕様（業務側と握った挙動）は `problem.md` に明記する**（一次ソース）
- `spec.md` は「設計の解説・代替案・観察ポイント」を担う（仕様の二重管理は避ける）
- `kata.test.ts` は `problem.md` の合意済み仕様と整合させる（テストが仕様の置き場にならないようにする）

## heat 構成

| heat | テーマ | 焦点 |
|------|--------|------|
| heat-1 | `createCache` | クロージャの基本: 状態の隠蔽と API の公開 |
| heat-2 | `createTTLCache` | 時間軸の追加: いつ無効化するかの設計判断 |
| heat-3 | `createAsyncCacheLoader` | DI と非同期: 依存性注入の意義を体感する |

## 実務との対応

- メモ化を伴う設定オブジェクトのキャッシュ
- API レスポンスの時限キャッシュ
- タブ・フィルタごとのデータ取得ゲート

これらはレガシー jQuery 環境でも、グローバル変数で書かれていることが多い。それを「クロージャに閉じ込めてテスト可能にする」のがこの forge のテーマ。
