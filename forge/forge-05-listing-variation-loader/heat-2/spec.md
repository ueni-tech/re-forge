# [heat-2] spec.md — 答え合わせ用

**注意: kata.ts を実装してから開くこと。**

## applyListingCodeParam の想定仕様

① `requestedCode` が `null` / `undefined` / 空文字のとき `defaultCode` をそのまま返す。

② `requestedCode` が `variationSkus` のいずれの `code` にも一致しないとき `defaultCode` を返す。

③ 一致するものがあれば `requestedCode` を返す。

※ `trim` は行わない。コードは完全一致で判定する。

## buildGoodsImageVariationsMap の想定仕様

① `variationSkus` の各 `code` に対して `{ path: baseUrl + code + ".html" }` を作り、`code` をキーとして `Record` に格納する。

② `code` が空文字のエントリはスキップする。

③ `variationSkus` が空なら空オブジェクト `{}` を返す。

## 自分の契約と比較する観点

### applyListingCodeParam

- **「無効なら無視して defaultCode を返す」の判断**: なぜ例外を投げないか? URLは外部入力で信頼できないから、不正値で落とすのではなく無視する方が安全。これを契約として書けたか?
- **trim しない判断**: 「優しさ」で trim したくなる気持ちと、「完全一致で判定する」厳格さのトレードオフ。仕様が後者を選んでいる理由を考える。

### buildGoodsImageVariationsMap

- **空文字 code のスキップ**: なぜマップに含めないか? キーが空文字だと取り出し時に `map[""]` という奇妙なアクセスになる。データ汚染を関数の内側で防いでいる例。
- **問題文に書かなかった2つの曖昧さ**:
  - 同じ code が複数あったら? → 後勝ち(`Object.assign` の挙動)。仕様には書かれていない暗黙ルール。
  - `baseUrl` の末尾スラッシュ無しは? → 単純な文字列連結なので、呼び出し側の責任。これも仕様に書かれていない。
  - **問題文に書いていない部分にこそ、設計判断の余地がある。**自分がどこまで気づけたか観察する。

## 観察ポイント

両関数とも「**呼び出し側の都合の悪い入力を、関数の内側で吸収する**」設計になっている。

- 不正なクエリ → 無視
- 空文字 code → スキップ
- 空配列 → 空マップ

これは heat-1 の「常に値を返す」と同じ思想。**「関数を使う側に try/catch や if 文を書かせない」**という設計判断のパターン。

このパターンは「呼び出し側に対する優しさ」だが、行き過ぎると「壊れたデータが見過ごされる」リスクもある。トレードオフがあることを認識する。
