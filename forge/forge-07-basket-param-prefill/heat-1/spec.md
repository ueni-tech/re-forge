# [heat-1] spec.md — 答え合わせ用

**注意: kata.ts を実装してから開くこと。**

## 想定仕様

① `el.type` が `"radio"` または `"checkbox"` で `el.checked === false` のとき: そのエントリは**スキップ**する。

② `el.dataset.basketParam` が `undefined` または `""` のとき: そのエントリは**スキップ**する。

③ 採用する値:
- `el.dataset.basketValue` が `undefined` でも `""` でもなければ `basketValue` を採用する
- そうでなければ `el.value` を採用する（フォールバック）

④ `type` が `"text"` などラジオ/チェックボックス以外のとき: `checked` の値に関係なく ②③ の判定だけで採用可否を決める。

⑤ 有効なエントリが 0 件のときは `{}` を返す（`null` や `undefined` ではない）。

## 自分の契約と比較する観点

### 「最小インターフェース型」で受ける設計

想定仕様の引数は `HTMLElement[]` ではなく、こう絞られている:

```ts
type CollectableElement = {
  type: string;
  checked: boolean;
  value: string;
  dataset: { basketParam?: string; basketValue?: string };
};
```

実コードでは `HTMLInputElement` などが渡るが、関数が**実際に触るプロパティだけ**を型として定義している。

- メリット: jsdom を立ち上げずに、プレーンオブジェクトでテストできる
- メリット: シグネチャを読むだけで「この関数は DOM のどこを触るか」が分かる
- 副次効果: DOM 以外の入力源（例: テスト用のモック、シリアライズされた状態）からも呼べる

これは「**依存を必要最小限の型で受ける**」設計パターン。型を絞ることで、関数の責務が型レベルで明示される。

### スキップ条件の順序

想定実装は概ねこの順序:

```
1. radio/checkbox なら checked をチェック → 未チェックならスキップ
2. basketParam が空ならスキップ
3. 値を決める（basketValue 優先、フォールバック el.value）
4. payload に書き込む
```

順序を入れ替えても**結果は変わらない**（早期 return の集合のため）。ただし、**読み手の認知負荷**は変わる:

- 「コストの低い判定 / よく落ちる条件」を先に書くと、コードを上から読みやすい
- 「テストで網羅されている条件」を先に書くと、書き手の意図が伝わりやすい

順序の正解はないが、「**自分はなぜこの順にしたか**」を言葉にできるかが学びどころ。

### `""` と `undefined` を同じ扱いにする判断

想定仕様は `basketValue === ""` を「未指定」として扱い、`el.value` にフォールバックする。

別の判断もあった:

- `basketValue === ""` を「明示的な空文字」として採用 → 空文字が payload に入る
- どちらも仕様として成立する

採用された判断の意図: 実コードで `data-basket-value=""` が書かれているのは「HTML テンプレートの都合で属性は出ているが、値はラジオの選択値（`el.value`）に任せたい」ケース。**運用上の現実**から「`""` は未指定とみなす」が選ばれている。

「**仕様は実装の都合だけでなく、運用の現実からも決まる**」という観察ポイント。

### 戻り値が `{}` であること

「該当なし」を表す方法は3つある:

| 選択肢 | 呼び出し側の扱い |
|--------|-----------------|
| `{}` を返す | `Object.keys(result).length` で件数判定。スプレッド可能 |
| `undefined` を返す | nullish チェックが必要 |
| 例外を投げる | try/catch が必要 |

想定仕様は `{}`。理由は「呼び出し側で `JSON.stringify(payload)` してそのまま sessionStorage に入れる」ような使い方を想定しているため。**「空でも常に同じ型を返す」と、呼び出し側の分岐が消える**。

これは「**呼び出し側に意識させない責務の引き受け方**」の一例。

## 観察ポイント: 純粋関数化の価値

実コードでは「DOM 走査 + payload 組み立て + sessionStorage 保存」がひとつの関数にまとまっていたとする。それを次のように分解する:

```
[DOM 走査] → [collectPayload (純粋関数)] → [sessionStorage 保存]
```

`collectPayload` だけ純粋関数として切り出すと:

- 入力 = `CollectableElement[]`、出力 = `Record<string, string>` という**型だけで責務が分かる**
- 副作用がない（DOM も storage も触らない）ため、テストが書きやすい
- 入力を作る側（DOM 走査）と出力を消費する側（sessionStorage）を、それぞれ独立に差し替えられる

レガシーコードを純粋関数化する第一歩は「**副作用を持つ呼び出しをシグネチャから追い出す**」こと。この heat はその最小単位の練習。

## 観察ポイント: 「触らないなら型にしない」原則

`HTMLInputElement` を全部受けず、`CollectableElement` という小さな型で受けることのもう一つの効果:

- もし将来「`el.name` も payload に含めたい」となったら、`CollectableElement` に `name` を**足す必要がある**
- 型の変更がそのまま「関数の責務が広がった」というシグナルになる

「**型が広いと、何が責務かぼやける**」。最小インターフェース型は、責務の輪郭を型で守る道具。
