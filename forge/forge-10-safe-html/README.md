# forge-10-safe-html — 文字列連結による HTML 生成の安全化

老舗EC のレガシーコードにありがちな「`'<li>' + name + '</li>'` で HTML を組み立てる XSS 温床コード」を題材に、安全な HTML 生成を3段階で身につける forge。

## この forge は競技プログラミング形式

forge-01〜09 とは異なり、この forge は **LeetCode / Codewars のような形式**で構成している。

- `spec.md` や契約4問はない。`problem.md` に **問題文・関数シグネチャ・入出力例・制約** だけを書く。
- 各 heat は独立した1問。入力 → 出力が明確に決まっており、テストを通せば正解。
- ファイルは `problem.md` / `kata.ts` / `kata.test.ts` / `kata.solution.ts` の4つ。
- `kata.ts` には**設計メモと「与えられた型」だけ**を置く。**関数シグネチャは渡さない**ので、`problem.md` を読んで引数・戻り値の型から自分で設計・宣言する（`export` 名はテストに合わせる）。最初はテストが落ちる状態から始まる。

「**仕様どおりに動くものを速く正確に書く**」訓練が主目的。

## 設計メモ（competitive 版の言語化装置）

forge-08 までの重い契約4問の代わりに、各 `kata.ts` の先頭に **どの kata でも使える汎用の設計問い（4問）** を「設計メモ」コメントとして埋め込んでいる。3 heat すべて同じ問い。

| # | 問い |
|---|------|
| Q1 | 入力の境界（空・0件・想定外の値）で何を返すと決めたか? |
| Q2 | 呼び出し側に何を保証し、何をしないと決めたか?（例外・破壊の有無） |
| Q3 | なぜこの書き方・データ構造を選んだか?（他の選択肢と比べて） |
| Q4 | この処理は他でも要りそうか?（切り出す単位は適切か） |

使い方は2通り。

- **解く前**: 念頭に置くチェックリストとして眺める。
- **解いた後**: 「こう考えたから、こう書いた」を `→` の後ろに1〜2行で残す。答えがある項目だけでよい。

参考回答は `kata.solution.ts` 末尾の「設計メモ 参考回答（汎用4問 × この kata）」にある。同じ4問にこの kata ならどう答えるかの例なので、自分の答えと突き合わせる。

「速く正確に書く」だけで終わらせず、**なぜその実装にしたかを一言で言える**ようにするのが狙い。

## heat 構成

| heat | 関数 | 難易度 | 焦点 |
|------|------|--------|------|
| heat-1 | `escapeHtml` | ★☆☆ | HTML 特殊文字5つの実体参照変換 |
| heat-2 | `buildAttributes` | ★★☆ | 属性文字列の組み立てと値のエスケープ |
| heat-3 | `renderProductList` | ★★★ | エスケープを組み合わせた一覧 HTML の生成 |

各 heat は独立しているので、heat-1 のエスケープ処理を後続でも自分で書き直す（標準ライブラリのように使い回す前提にしない）。

## 進め方

```bash
npx vitest forge-10-safe-html          # forge 全体
npx vitest forge-10-safe-html/heat-1   # heat 単体
```

`kata.ts` を実装してテストを通す。行き詰まったら `kata.solution.ts` を見る。

## 実務との対応

- 商品名・カテゴリ名・レビュー本文など、ユーザー由来の文字列を HTML に差し込む全箇所
- `href` / `title` / `data-*` 属性へ動的な値を入れる箇所（属性ブレイクアウト型 XSS の入口）

レガシー jQuery 環境では `$el.html('<li>' + data.name + '</li>')` が典型的な危険コード。「**HTML に文字列を入れる前に必ずエスケープを通す**」習慣を、純粋関数として手に馴染ませるのがこの forge のテーマ。
