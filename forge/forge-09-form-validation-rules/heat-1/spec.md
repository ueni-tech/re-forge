## 仕様の一次ソース

この heat の「合意済み仕様（握る挙動）」は `problem.md` に明記されている。

## 自分の契約と比較する観点

### 「未入力」の判定

- `required` は `value === ""` だけでなく `value.trim() === ""` で判定したか?
- 「スペースだけの名前」は実フォームで頻繁に起きる。`trim` をどこでかけるかは設計判断 — `required` の**内側**でかければ、呼び出し側は trim を意識しなくてよい。
- 逆に `maxLength` で trim をかけると「空白込みで50文字」と「trim後50文字」のどちらを数えているのか曖昧になる。**trim するのは required だけ**、という非対称に気づけたか。

### `maxLength` / `pattern` が空文字を OK にする設計

- 「空文字は OK」に違和感を覚えたかもしれない。これは **「未入力チェックは required だけの責務」** という責務分離の表れ。
- もし `pattern` が空文字を NG にしたら、「電話番号は任意。ただし入力するなら形式チェック」という要件が表現できなくなる。
- 必須にしたいフィールドには `required` と `pattern` を**両方**付ける。ルールを直交させておけば、組み合わせで要件を表現できる。これが heat-2 の合成につながる。

### boolean ではなく結果型を返す設計

- `(value) => boolean` だと、NG のとき**何を表示するか**を呼び出し側が別途管理することになる。レガシーコードで `if (!checkTel(tel)) { alert('電話番号が...') }` とメッセージが呼び出し側に散らばるのはこれが原因。
- `{ ok: false, message }` なら、**ルールとメッセージが一体**で持ち運べる。判定結果と表示文言の対応がずれる事故を型レベルで防げる。
- この形は TypeScript の **discriminated union（判別可能なユニオン型）**。`if (!result.ok)` で絞り込むと `result.message` に安全にアクセスできる。

## 観察ポイント: ファクトリがバリデータを返す2段構え

```ts
function maxLength(max: number, message: string): Validator {
  return (value) => value.length <= max
    ? { ok: true }
    : { ok: false, message };
}
```

`max` と `message` はクロージャに閉じ込められ、返ってきた関数は `(value) => ValidationResult` という**同じ型**になる。

これが重要なのは、**設定の異なるルールたちを同じ型の配列に並べられる**から:

```ts
const nameRules: Validator[] = [
  required("お名前を入力してください"),
  maxLength(50, "お名前は50文字以内で入力してください"),
];
```

レガシーコードの if-else は「ルールがコードの構造に埋まっている」状態。ファクトリで関数化すると「ルールがデータとして並ぶ」状態になる。この差が heat-2（合成）・heat-3（スキーマ駆動）の土台になる。

forge-01 の `createCache` が「状態」をクロージャに閉じたのに対し、ここでは「設定」を閉じている。閉じるものが違うだけで、技法は同じ。

## 観察ポイント: 副作用を持たないことの価値

合意済み仕様の「alert・DOM操作・ログを持たない」は、消極的な制約ではない。

- alert を内蔵した瞬間、このバリデータは **jsdom なしでテストできなくなり**、「エラーをまとめて表示したい」という将来要件にも対応できなくなる。
- 「判定する」と「通知する」を分けておけば、通知方法（alert / インライン表示 / submit ボタンの無効化）を後から自由に選べる。

神関数の問題は行数ではなく、**判定と通知が分離できない**ことにある。
