## 自分の契約と比較する観点

### 打ち切り（fail-fast）に意味があるか

- `composeValidators` の打ち切りは単なる効率の話ではない。**ルールの並び順が「メッセージの優先順位」を表現する**。
- `[required("必須です"), pattern(..., "形式が不正です")]` の順なら、空文字のとき「必須です」だけが出る。もし全部実行して最初のメッセージを選ぶ実装でも結果は同じだが、「後続を実行しない」と握っておくと、将来コストの高いバリデータ（サーバー照合など）を混ぜても安心できる。
- heat-1 で `pattern` が空文字を OK にした設計のおかげで、実は順序を入れ替えても「空文字で形式エラーが出る」事故は起きない。**ルールの直交性が、合成の自由度を生んでいる**。

### 空配列 = 常に OK

- 「ルールが1つもない」は「制約がない」ということ。制約がなければ何でも通る、と解釈するのが自然。
- これは「全称量化の空集合は真（vacuous truth）」という論理の話でもあるが、実務的には **heat-3 でルール未設定のフィールドがあってもガード不要になる** という伏線。

### `collectErrors` が `Validator` を返せない理由

- `ValidationResult` の NG 側は `message: string` を**1つ**しか持てない。エラーが複数あるとき、この型には載らない。
- つまり「全収集」をやりたければ、**戻り値の型から変える必要がある**。型がアーキテクチャの分岐点になっている例。
- 代替案として `{ ok: false, messages: string[] }` のような型を新設する手もある。今回 `string[]` を直接返す形にしたのは、「空配列なら OK」という判定が `errors.length === 0` の1行で済み、追加の型が要らないから。

## 観察ポイント: 入力と出力が同じ型である価値

```ts
const nameValidator = composeValidators([
  required("お名前を入力してください"),
  maxLength(50, "お名前は50文字以内で入力してください"),
]);
// nameValidator は Validator 型。単体の required と区別がつかない
```

合成結果が `Validator` 型そのものなので、

- 合成済みバリデータを**さらに別の合成の材料にできる**（共通ルールセット + フィールド固有ルール、のような積み上げ）
- 受け取る側（heat-3 のスキーマ）は「これが単体ルールか合成済みか」を**知らなくてよい**

この「閉じた合成（closed composition）」は、jQuery のチェーン、Promise の `.then`、配列の `map` にも共通する性質。**同じ型に戻ってくるから無限につなげられる**。

## 観察ポイント: 戦略の分離

神関数では「打ち切るかどうか」が `return false` というコードの構造に焼き付いていた。表示要件が「全件まとめて表示」に変わったら、関数全体を書き直すしかない。

heat-2 では、

- ルール（heat-1 のバリデータ）は**1行も変えずに**
- 実行戦略だけを `composeValidators` / `collectErrors` で選べる

「何を検証するか」と「どう実行するか」が分離されているから、片方の変更がもう片方に波及しない。これが責務分離の実利。
