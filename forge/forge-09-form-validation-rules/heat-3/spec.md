# [heat-3] spec.md — 答え合わせ用

**注意: kata.ts を実装してから開くこと。**

## 仕様の一次ソース

この heat の「合意済み仕様（握る挙動）」は `problem.md` に明記されている。
`spec.md` は仕様そのものではなく、設計の観察ポイント・判断の背景・代替案の比較を担う。

## 自分の契約と比較する観点

### 「フィールド内は打ち切り・フィールド間は全検証」のハイブリッド

- フィールド内打ち切り: 1つの入力欄に「必須です」と「形式が不正です」を同時に出しても、ユーザーは混乱するだけ。**1欄につきエラーは1つ**。
- フィールド間全検証: 1個直すたびに submit して次のエラーが出る「もぐらたたき」は最悪の UX。**全欄のエラーを一度に見せる**。
- つまりこのハイブリッドは実装都合ではなく、**表示単位（フィールド）と修正単位（フォーム全体）の違い**から導かれた仕様。神関数の `return false` 連打は後者を構造的に実現できない（最初の NG で関数ごと終わるから）。

### `when` に `values` 全体が渡される理由

- 「配送先が『別住所』のとき住所必須」のように、条件付きルールの条件は**ほぼ必ず他のフィールド**を見る。自フィールドの値で条件分岐したいなら、それはバリデータ自身の仕事（heat-1 の `pattern` が空文字を OK にしたのがその例）。
- `when` は「フィールド横断の条件」専用の置き場、と整理できる。

### 欠損キーを `""` 扱いにする理由

- HTML フォームでは、チェックされていないチェックボックスや disabled な input は**値ごと送信されない**。「キーが無い」は実フォームでは正常系。
- もし `values[field.name]` を `undefined` のままバリデータに渡すと、heat-1 のバリデータは `string` を前提にしているので `value.trim()` で実行時エラーになる。**境界（validateForm）で正規化して、内側（バリデータ）は正常な入力だけを扱う**という防衛ラインの引き方。

### name 重複時の挙動

- 仕様に書かれていない以上、自分の実装がどちらに転ぶかを**把握して契約に書く**ことが大事。素直にループで `errors[field.name] = message` と代入していれば後勝ちになる。
- 「仕様に無い挙動は、せめて自分が知っている状態にする」— 未定義動作を放置しないのが契約を書く目的。

### `composeValidators` を再利用したか

- heat-2 の `composeValidators` は「最初の NG で打ち切り、その結果を返す」をすでに実装済み。再利用すれば validateForm のループ本体は数行になる。
- 直接ループを書いても正しく動くが、「打ち切り戦略」のロジックが2箇所に存在することになる。どちらを選んだにせよ、**重複に気づいた上で選んだか**が問われる。

## 観察ポイント: ルールがデータになると何が起きるか

神関数とスキーマで「フォームにフィールドを1つ追加する」作業を比べる。

| | 神関数 | スキーマ駆動 |
|---|---|---|
| 追加方法 | if ブロックをコピペして編集 | スキーマに1行追加 |
| ルール全体の把握 | 関数を全部読む | スキーマを眺める |
| 検証ロジックの変更 | 全 if ブロックに波及 | `validateForm` 1箇所 |
| テスト | DOM ごと再現が必要 | 値のオブジェクトを渡すだけ |

スキーマは「設定」であり、`validateForm` は「エンジン」。フィールドが10個増えてもエンジンは1行も変わらない。**変更頻度の高いもの（ルール）と低いもの（実行方法）を分けて、頻度の高い方をデータに降ろす**のがこのパターンの核心。

forge-07 heat-3 の Strategy テーブル（input タイプ → 適用関数）と同じ発想で、あちらは「分岐をテーブルに」、こちらは「ルールの列挙をテーブルに」降ろしている。

## 観察ポイント: レガシー環境への持ち帰り方

この forge は TypeScript + vitest で書いたが、構造自体は jQuery 環境にそのまま持ち帰れる。

```js
// 検証エンジンとスキーマは純粋な JS。jQuery 非依存
var errors = validateForm(collectFormValues($form), orderFormSchema);

// DOM への反映だけ jQuery で行う
Object.keys(errors).forEach(function (name) {
  $('[name="' + name + '"]').addClass('is-error');
});
```

ポイントは **「DOM から値を集める」「検証する」「結果を表示する」を別の関数にする**こと。真ん中の「検証する」だけがこの forge の成果物であり、そこが純粋関数ならレガシー環境でもテストが書ける。
