# forge-11-lgom-layout-parser — 印面内容フィルタタグのパース

住所印商品一覧ページの「印面内容」フィルタ機能を題材に、`lGomLayout` フィールドの生 HTML 文字列をフィルタタグ配列へ変換する処理を3段階で実装する forge。

実務コードの直接抽出: `_pkg/libs/AddressInmenContentParser.php`

## この forge は競技プログラミング形式

LeetCode / Codewars のような形式で構成している（forge-09・forge-10 と同じ）。

- `problem.md` に **問題文・入出力例・合意済み仕様・実装する関数名** だけを書く（関数シグネチャは渡さない）。
- 各 heat は独立した1問。入力 → 出力が明確に決まっており、テストを通せば正解。
- ファイルは `problem.md` / `kata.ts` / `kata.test.ts` / `kata.solution.ts` の4つ。
- `kata.ts` には**設計メモと「ここに実装する」コメントのみ**を置く。最初はテストが落ちる状態から始まる。

「**仕様どおりに動くものを速く正確に書く**」訓練が主目的。

## 元ネタの実務コード

```php
// _pkg/libs/AddressInmenContentParser.php
// lGomLayout（例: "1行目：社名or店名<br>2行目：役職＆氏名"）を
// フィルタタグ配列（例: ["会社名・店名", "氏名"]）に変換するパーサー。
// 住所印一覧の印面内容フィルタ用 data-filter-inmen-content 属性に埋め込む。
//
// MVP では同じロジックを JS（address-list-filter.js）に書いていた。
// 本番化でサーバー描画時（PHP）に寄せ、JS パーサーを削除した。
```

## 設計メモ（competitive 版の言語化装置）

各 `kata.ts` の先頭に **どの kata でも使える汎用の設計問い（4問）** を「設計メモ」コメントとして埋め込んでいる。3 heat すべて同じ問い。

| # | 問い |
|---|------|
| Q1 | 入力の境界（空・0件・想定外の値）で何を返すと決めたか? |
| Q2 | 呼び出し側に何を保証し、何をしないと決めたか?（例外・破壊の有無） |
| Q3 | なぜこの書き方・データ構造を選んだか?（他の選択肢と比べて） |
| Q4 | 前後の処理との役割分担は適切か?（この関数が担うこと・担わないこと） |
| メモ | Q1〜Q4 に当てはまらない気づき・判断根拠（空欄可） |

使い方は2通り。

- **解く前**: 念頭に置くチェックリストとして眺める。
- **解いた後**: 「こう考えたから、こう書いた」を `→` の後ろに1〜2行で残す。

参考回答は `kata.solution.ts` 末尾の「設計メモ 参考回答（汎用4問 × この kata）」にある。

## heat 構成

| heat | 関数 | 難易度 | 焦点 |
|------|------|--------|------|
| heat-1 | `extractRowLabel` | ★☆☆ | 正規表現2本で行文字列からラベルを取り出す |
| heat-2 | `rowLabelToFilterTags` | ★★☆ | ルールテーブルでラベル→タグ配列へ変換する。`TEL＆FAX` の2タグ特例 |
| heat-3 | `parseLGomLayout` | ★★★ | 生 HTML 文字列をパイプラインで変換し、重複なしのタグ配列を返す |

各 heat は独立しているので、ヘルパー関数を後続で使い回す前提にしない。

## 進め方

```bash
npx vitest forge-11-lgom-layout-parser          # forge 全体
npx vitest forge-11-lgom-layout-parser/heat-1   # heat 単体
```

`kata.ts` を実装してテストを通す。行き詰まったら `kata.solution.ts` を見る。

## 実務との対応

- `data-filter-inmen-content` 属性値の生成（住所印一覧の印面内容フィルタ）
- 「CMS のラベル文字列」と「フィルタ UI のボタン値」は表記が異なる。この変換テーブルが両者を繋ぐ
- ルールが追加・変更されたとき「どこを1行直せばよいか」がすぐ分かる構造が求められる
